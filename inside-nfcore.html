Inside nf-core/rnaseq

A plain-language view of what the pipeline does after you give it FASTQ files and a samplesheet.

FASTQ input

What it does

The pipeline starts from FASTQ files. Each row in the samplesheet tells nf-core/rnaseq which FASTQ files belong to each biological sample and whether the reads are single-end or paired-end.

What it produces

This stage does not create final biological results. It validates input paths and prepares the files for the next steps.

Why you should care

If a FASTQ path, sample name, or mate-pair relationship is wrong, the pipeline may fail early or produce outputs assigned to the wrong sample.

Raw read QC

What it does

Raw read QC checks sequencing quality before trimming or alignment. Tools such as FastQC summarize base quality, adapter content, sequence duplication, GC content, and overrepresented sequences.

What it produces

FastQC HTML reports and summary files for each FASTQ file.

Why you should care

Raw QC tells you whether the sequencing data look reasonable before the pipeline changes anything. It can reveal poor quality reads, adapter contamination, or unusual library composition.

Trimming

What it does

Trimming removes adapter sequence and low-quality read ends when trimming is enabled. This can improve alignment or quantification for libraries with obvious adapter contamination.

What it produces

Trimmed FASTQ files and trimming reports, depending on the selected pipeline options.

Why you should care

Trimming changes the reads used downstream. You should know whether trimming was enabled when comparing runs or explaining the analysis.

Alignment or quantification

What it does

nf-core/rnaseq can align reads to a genome, quantify transcripts, or do both depending on the selected options. Common choices include STAR for genome alignment and Salmon for transcript quantification.

What it produces

Alignment files and logs for genome-aligned runs, Salmon `quant.sf` files for transcript quantification, and mapping or assignment summaries.

Why you should care

This stage is usually compute-heavy and sensitive to reference choice. Memory errors often happen here, especially during STAR alignment or genome indexing.

Count generation

What it does

The pipeline produces gene-level or transcript-level count-like tables that can be used for downstream differential expression and expression summaries.

What it produces

Count matrices, Salmon quantification files, and related feature or transcript summaries. Exact filenames depend on the pipeline version and options.

Why you should care

These are the files most downstream tools need. DESeq2 and edgeR use count-based inputs, while visualization and reporting often use transformed or normalized summaries later.

MultiQC summary

What it does

MultiQC collects many small QC reports into one readable HTML report. It combines FastQC, trimming, alignment, quantification, and other tool summaries.

What it produces

`multiqc_report.html` and supporting MultiQC data files.

Why you should care

MultiQC is usually the first report to open after a run. It helps you spot outlier samples, failed steps, low mapping rates, high duplication, or unexpected library behaviour.

Pipeline metadata

What it does

nf-core/rnaseq records the pipeline version, parameters, software versions, execution reports, timelines, and trace information.

What it produces

`pipeline_info/`, `software_versions.yml`, `configs/configure.yaml`, `execution_report.html`, `timeline.html`, and trace files.

Why you should care

Metadata makes the run auditable and reproducible. Keep these files with your final results so you can explain exactly how the analysis was produced.
