nf-core/rnaseq Pipeline

This guide targets nf-core/rnaseq v3.26.0. Every run is pinned to this version with nextflow run nf-core/rnaseq -r 3.26.0 (set via pipeline_version in configs/configure.yaml), so the commands, parameters, and outputs described here match the official 3.26.0 documentation.

nf-core/rnaseq is the official nf-core RNA-seq analysis pipeline. It is built with Nextflow and is designed for RNA-seq data from organisms with a reference genome and annotation. It can start from FASTQ files or compatible BAM inputs, then run quality control, trimming, alignment or pseudo-alignment, quantification, MultiQC reporting, and pipeline metadata capture.

Inputs

FASTQ files + samplesheet + reference genome + annotation

Pipeline

QC → trimming → STAR + Salmon or Salmon-only → count matrix → MultiQC report

Outputs

gene/transcript counts, quality-control reports, logs, software versions, and reproducibility metadata.

QC

Checks raw read quality before analysis. This helps you see read quality, adapter content, duplication, overrepresented sequences, and other warning signs before trusting downstream results.

Trimming

Removes adapters and low-quality sequence when trimming is enabled. This can improve mapping or quantification when raw reads contain technical sequence or poor-quality tails.

STAR + Salmon

Aligns reads to the reference genome with STAR, then uses Salmon for transcript-level quantification. This mode is useful when you want alignment evidence, mapping metrics, and a full reference-guided RNA-seq run.

Salmon-only

Quantifies reads without genome alignment. This is faster and lighter, but it is mainly for quantification-focused runs and does not produce the same genome-alignment evidence as STAR-based mode.

Count matrix

Produces gene-level and/or transcript-level count or abundance tables for downstream analysis. These are the key files used later for differential expression, visualisation, and other RNA-seq summaries.

MultiQC report

Collects QC and pipeline metrics into one HTML report. Open this first after a run to review sequencing quality, trimming, mapping or quantification performance, and whether the run looks usable.

This guide is not a replacement for the official nf-core/rnaseq documentation. It is a beginner-friendly HPC wrapper that helps users prepare inputs, validate configuration, run safely on HPC, resume failed jobs, and download results.

What Each Step Does

  1. Initial Project: start from one reusable project template.
  2. Prepare & Validate Inputs: create samplesheet and validate FASTQ, reference, annotation, and project files locally.
  3. Pipeline Configure: set simple run choices in configs/configure.yaml.
  4. Upload to HPC: transfer the prepared project folder with resume-safe rsync.
  5. Preflight Validation: run Smoke Test and One Sample Run on HPC before the full analysis.
  6. Run & Monitor: launch the full nf-core/rnaseq workflow and monitor progress.
  7. Download Results: retrieve results, reports, logs, MultiQC outputs, and summaries locally.

Choose Pipeline Mode

Use STAR + Salmon when you want genome alignment outputs, splice-aware mapping evidence, and the most complete nf-core/rnaseq run. This is usually the safer default for a new species project when a reference genome and annotation are available.

Use Salmon only when you mainly need transcript or gene quantification and want a faster, lighter run. In nf-core/rnaseq this uses --pseudo_aligner salmon with --skip_alignment, rather than --aligner salmon.

Resource settings are estimated from the chosen mode, species, genome size, and sample count. Large genomes and many samples generally need more memory, CPU, and walltime, especially for STAR-based alignment.

Salmon GC Bias Correction (Important)

By default, nf-core/rnaseq runs Salmon without GC bias correction. The DESeq2 authors explicitly recommend enabling it:

"We recommend using the --gcBias flag which estimates a correction factor for systematic biases commonly present in RNA-seq data (Love, Hogenesch, and Irizarry 2016; Patro et al. 2017), unless you are certain that your data do not contain such bias."

Because this guide otherwise uses pipeline defaults, it turns this DESeq2-recommended option on by default: configure.sh writes gc_bias: true, and the run scripts pass --extra_salmon_quant_args=--gcBias to Salmon. It is safe to leave on; set gc_bias: false in configs/configure.yaml only if you have a specific reason to disable it.

Reference: DESeq2 vignette — Salmon/tximport quantification.

Smoke Test

The Smoke Test checks whether the HPC environment can run nf-core/rnaseq at all. It uses tiny built-in test data, not user samples.

It checks Nextflow, Apptainer/Singularity, container downloads, cache paths, permissions, modules, work directory, and scratch directory.

Question answered: Can this HPC + nf-core environment run?

When to run it: generally only the first time you run on a new HPC environment. Once the smoke test passes on that cluster, you can skip it on later projects and go straight to the One Sample Run.

One Sample Run

The One Sample Run checks whether the current project configuration can survive a real run. It performs dry-run/config validation, then runs one real sample.

The default sample is the largest sample, because the goal is resource validation as well as workflow validation.

Auto-fix and Resume

Safe infrastructure and resource fixes may be applied automatically when the user allows it, such as creating cache directories, fixing container/image cache paths, moving work or temporary directories, increasing memory or walltime limits, and resuming after the fix. Biological settings are never changed automatically.

Biological settings must never be changed automatically: reference genome, annotation, strandedness, aligner choice, metadata, sample grouping, or experimental design.

Troubleshooting Notes

  • STAR killed: check memory and walltime; use One Sample Run to estimate resources.
  • Container download failure: check Apptainer/Singularity cache path and permissions.
  • Disk quota issue: move workdir to scratch or clean old work/results folders.
  • Samplesheet error: rerun make_samplesheet and input validation.
  • Pipeline stuck: run monitor_run.sh and inspect recent Nextflow logs.

After Downloading Results

Downloading results finishes the nf-core/rnaseq run, but it is usually the start of biological analysis.

What next? Review QC first, then use the count or quantification outputs for differential expression, enrichment, co-expression, visualisation, Shiny apps, or reporting.

  • Open MultiQC first and check whether sequencing and mapping quality look acceptable.
  • Use count or quantification outputs for differential expression analysis.
  • Use reproducibility files when sharing or rerunning the project.
  • Move into enrichment, co-expression, visualisation, Shiny, or reporting only after QC looks reasonable.

FAQ

What is nf-core/rnaseq?

A community-maintained RNA-seq pipeline built with Nextflow.

What is a samplesheet?

A CSV file that maps sample IDs to FASTQ files and strandedness.

What is Apptainer?

A container runtime commonly used on HPC clusters.

Can I resume a failed run?

Yes. Keep the work directory and use Nextflow resume.

What should I keep after the run?

Results, reports, logs, MultiQC, quantification outputs, pipeline metadata, samplesheet, configs, scripts, and reference metadata.
