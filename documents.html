nf-core/rnaseq Pipeline

This guide targets nf-core/rnaseq v3.26.0. Every run is pinned to this version with nextflow run nf-core/rnaseq -r 3.26.0 (set via pipeline_version in configs/configure.yaml), so the commands, parameters, and outputs described here match the official 3.26.0 documentation.

nf-core/rnaseq is the official nf-core RNA-seq analysis pipeline. It is built with Nextflow and is designed for RNA-seq data from organisms with a reference genome and annotation. It can start from FASTQ files or compatible BAM inputs, then run quality control, trimming, alignment or pseudo-alignment, quantification, MultiQC reporting, and pipeline metadata capture.

Inputs

FASTQ files + samplesheet + reference genome + annotation

Pipeline

QC → trimming → STAR + Salmon or Salmon-only → count matrix → MultiQC report

Outputs

gene/transcript counts, quality-control reports, logs, software versions, and reproducibility metadata.

QC

Checks raw read quality before analysis. This helps you see read quality, adapter content, duplication, overrepresented sequences, and other warning signs before trusting downstream results.

Trimming

Removes adapters and low-quality sequence when trimming is enabled. This can improve mapping or quantification when raw reads contain technical sequence or poor-quality tails.

STAR + Salmon

Aligns reads to the reference genome with STAR, then uses Salmon for transcript-level quantification. This mode is useful when you want alignment evidence, mapping metrics, and a full reference-guided RNA-seq run.

Salmon-only

Quantifies reads without genome alignment. This is faster and lighter, but it is mainly for quantification-focused runs and does not produce the same genome-alignment evidence as STAR-based mode.

Count matrix

Produces gene-level and/or transcript-level count or abundance tables for downstream analysis. These are the key files used later for differential expression, visualisation, and other RNA-seq summaries.

MultiQC report

Collects QC and pipeline metrics into one HTML report. Open this first after a run to review sequencing quality, trimming, mapping or quantification performance, and whether the run looks usable.

What Each Step Does

  1. Initial Project: start from one reusable project template.
  2. Prepare & Validate Inputs: create samplesheet and validate FASTQ, reference, annotation, and project files locally.
  3. Pipeline Configure: set simple run choices in configs/configure.yaml.
  4. Upload to Phoenix: transfer the prepared project folder with resume-safe rsync.
  5. Preflight Validation: run Smoke Test and One Sample Run on Phoenix before the full analysis.
  6. Run & Monitor: launch the full nf-core/rnaseq workflow and monitor progress.
  7. Download Results: retrieve results, reports, logs, MultiQC outputs, and summaries locally.

Choose Pipeline Mode

Use STAR + Salmon (star_salmon) when you want genome alignment outputs, splice-aware mapping evidence, and the most complete nf-core/rnaseq run. This is usually the safer default for a new species project when a reference genome and annotation are available. star_salmon provides both expression quantification and genome-aligned BAM files for potential downstream analyses such as SNP calling.

Use Salmon only when you mainly need transcript or gene quantification and want a faster, lighter run. In nf-core/rnaseq this uses --pseudo_aligner salmon with --skip_alignment, rather than --aligner salmon.

nf-core/rnaseq v3.26.0 offers four genome aligners via --aligner:

  • star_salmon — STAR (splice-aware) + Salmon quantification. The default and recommended choice for eukaryotes.
  • star_rsem — STAR alignment + RSEM quantification.
  • hisat2 — HISAT2 splice-aware alignment.
  • bowtie2_salmon — Bowtie2 + Salmon. Bowtie2 is splice-unaware: use this only for prokaryotic RNA-seq; it is not recommended for eukaryotic data.

Resource settings are estimated from the chosen mode, species, genome size, and sample count. Large genomes and many samples generally need more memory, CPU, and walltime, especially for STAR-based alignment.

Salmon GC Bias Correction (Important)

By default, nf-core/rnaseq runs Salmon without GC bias correction. The DESeq2 authors explicitly recommend enabling it:

"We recommend using the --gcBias flag which estimates a correction factor for systematic biases commonly present in RNA-seq data (Love, Hogenesch, and Irizarry 2016; Patro et al. 2017), unless you are certain that your data do not contain such bias."

Because this guide otherwise uses pipeline defaults, it turns this DESeq2-recommended option on by default: configure.sh writes gc_bias: true, and the run scripts pass --extra_salmon_quant_args=--gcBias to Salmon. It is safe to leave on; set gc_bias: false in configs/configure.yaml only if you have a specific reason to disable it.

Reference: DESeq2 vignette — Salmon/tximport quantification.

Samplesheet Format (nf-core/rnaseq)

nf-core/rnaseq reads a CSV samplesheet with this exact header:

sample,fastq_1,fastq_2,strandedness
  • sample — sample name. Use only letters, numbers, dot, underscore, or hyphen (no spaces or commas). Rows that share the same sample are treated as technical replicates and are concatenated by the pipeline.
  • fastq_1 — path to the R1 FASTQ. Must be gzipped (.fastq.gz or .fq.gz).
  • fastq_2 — path to the R2 FASTQ (gzipped). Leave empty for single-end data.
  • strandedness — one of forward, reverse, unstranded, or auto (with auto, Salmon infers strandedness per sample).

Example — the second row is a technical replicate of the first (same sample name), and the third row is single-end (empty fastq_2):

sample,fastq_1,fastq_2,strandedness
CONTROL_REP1,input/fastq/AEG588A1_S1_L002_R1_001.fastq.gz,input/fastq/AEG588A1_S1_L002_R2_001.fastq.gz,auto
CONTROL_REP1,input/fastq/AEG588A1_S1_L003_R1_001.fastq.gz,input/fastq/AEG588A1_S1_L003_R2_001.fastq.gz,auto
TREATMENT_REP1,input/fastq/AEG588A4_S4_L003_R1_001.fastq.gz,,auto

Step 2 generates this file with make_samplesheet.sh and checks it with input_validate.sh (header, unique/valid sample names, paired FASTQ existence, strandedness whitelist). Still open the file and confirm sample names and R1/R2 pairing look correct before uploading. Full details: official nf-core/rnaseq usage docs.

Smoke Test

The Smoke Test checks whether the Phoenix environment can run nf-core/rnaseq at all. It uses tiny built-in test data, not user samples.

It checks Nextflow, Apptainer/Singularity, container downloads, cache paths, permissions, modules, work directory, and scratch directory.

Question answered: Can this Phoenix + nf-core environment run?

When to run it: generally only the first time you run on a new Phoenix environment. Once the smoke test passes on Phoenix, you can skip it on later projects and go straight to the One Sample Run.

One Sample Run

The One Sample Run checks whether the current project configuration can survive a real run. It performs dry-run/config validation, then runs one real sample.

The default sample is the largest sample, because the goal is resource validation as well as workflow validation.

Auto-fix and Resume

Safe infrastructure and resource fixes may be applied automatically when the user allows it, such as creating cache directories, fixing container/image cache paths, moving work or temporary directories, increasing memory or walltime limits, and resuming after the fix. Biological settings are never changed automatically.

Biological settings must never be changed automatically: reference genome, annotation, strandedness, aligner choice, metadata, sample grouping, or experimental design.

After Downloading Results

Downloading results finishes the nf-core/rnaseq run, but it is usually the start of biological analysis.

What next? Review QC first, then use the count or quantification outputs for differential expression, enrichment, co-expression, visualisation, Shiny apps, or reporting.

  • Open MultiQC first and check whether sequencing and mapping quality look acceptable.
  • Use count or quantification outputs for differential expression analysis.
  • Use reproducibility files when sharing or rerunning the project.
  • Move into enrichment, co-expression, visualisation, Shiny, or reporting only after QC looks reasonable.

Preflight: dry run and sign-off checklist

scripts/smoke_run.sh already runs these preflight checks and the -preview dry run automatically at its start (before the tiny test run), so you normally do not run them by hand. This section documents what it checks, and the manual commands if you want to verify anything yourself.

Before committing a long, expensive run (the oat run was ~12 h and ~530 CPU-hours), do a cheap dry run and a quick sign-off. The simplest manual dry run is Nextflow's -preview, which builds the workflow and validates inputs without submitting any heavy jobs:

# quick check on a login node — no heavy jobs submitted
nextflow run nf-core/rnaseq -r 3.26.0 -profile apptainer -preview \
  -c nextflow.config \
  --input samplesheets/samplesheet.csv --outdir results/preview_test \
  --fasta genome.fasta.gz --gff genes.gff3.gz --aligner star_salmon --igenomes_ignore

Then run these on the HPC before launching or resuming, and confirm each item:

module list
echo $NXF_APPTAINER_CACHEDIR && du -sh $NXF_APPTAINER_CACHEDIR 2>/dev/null
du -sh work results references fastq 2>/dev/null
df -h .
squeue -u $USER
du -sh references/prebuilt_indexes/* 2>/dev/null
  • STAR index plan is clear — reuse the existing index, or rebuild intentionally with --save_reference.
  • Apptainer cache variable is set and populated.
  • Storage has enough free space (keep work/ on /scratchdata1).
  • STAR memory is high enough (≥130 GB for a large genome).
  • A BEDTOOLS_GENOMECOV_COMBINED memory override is present.
  • Library type (strandedness) is correct — keep it stable between runs.
  • work/ is kept if you intend to -resume.
  • Logs are saved before any major change.
  • Final outputs will be checked before any cleanup (see Troubleshooting #8).

Reusing reference indexes (large genomes)

For a large genome (e.g. hexaploid oat), building the STAR and Salmon indexes is the most expensive part of the run — STAR's index can be ~100 GB and needs the high-memory queue (~300 GB RAM). You do not need to rebuild it every run. If you already have prebuilt indexes, point the pipeline at them so they are reused instead of regenerated:

--star_index   /path/to/references/prebuilt_indexes/star \
--salmon_index /path/to/references/prebuilt_indexes/salmon

Rebuild only when the FASTA or annotation changed, the index is missing or possibly corrupted, or you want a fully clean run. When you do rebuild, pass --save_reference so the new index is written into the results and can be reused next time, and record which FASTA and GFF3/GTF were used. Indexes are large and easy to leave behind in a random work folder, so decide reuse-vs-rebuild deliberately and check du -sh references/prebuilt_indexes/* first.

Troubleshooting: known issues on large-genome HPC runs

These are real failures hit on a large-genome (oat) nf-core/rnaseq run on Phoenix, with the fix for each. They are the most likely things to go wrong on a similar run.

  1. STAR index rebuilt by accident. Huge and slow. Reuse via --star_index/--salmon_index, or rebuild on purpose with --save_reference; check du -sh references/* first.
  2. Container download timed out (often the Salmon image). Check the cache before running (echo $NXF_APPTAINER_CACHEDIR, du -sh it); if a run fails while pulling, fix the cache — don't start editing the pipeline command.
  3. Storage / quota filled mid-run. Big consumers are work/, the STAR index, BAMs. Check df -h . and du -sh work results references before starting/resuming; keep work/ on /scratchdata1.
  4. STAR ran out of memory. Observed ~112–114 GB MaxRSS; request ~130–200 GB. Confirm a kill with sstat -j JOBID.batch --format=JobID,MaxRSS,MaxVMSize before rerunning everything.
  5. BEDTOOLS_GENOMECOV_COMBINED out of memory. Needs a process-specific memory override (well above default). Confirm the override is in your config; if it fails again it is almost certainly the same issue.
  6. Strandedness. The oat run was unstranded (with Salmon/RSeQC evidence). Don't switch to forward/reverse without new evidence — it changes the counts and the DESeq2 results.
  7. work/ and -resume. Keep work/ to resume; add -resume only when inputs are unchanged (same samplesheet, FASTA, annotation, config logic, work dir). If anything changed, treat it as a new run.
  8. SLURM going quiet ≠ finished. An empty squeue does not mean all outputs are written. Verify the real outputs — e.g. find results -name "quant.sf" | wc -l (expect one per sample) — before deleting anything.

Before editing configs, deleting files, or resuming, snapshot the state first: cp .nextflow.log logs/nextflow_$(date +%Y%m%d_%H%M).log and squeue -u $USER > logs/squeue_$(date +%Y%m%d_%H%M).txt, then triage with grep -i "error\|killed" .nextflow.log | tail.

FAQ

Why use sbatch instead of bash?

The Phoenix login node is shared by everyone, so heavy work run there with bash script.sh will be throttled or killed. sbatch script.sh hands the job to the SLURM scheduler, which runs it on a compute node with the CPUs, memory, and time you requested. Use bash only for tiny, instant helpers (input_validate.sh, monitor_run.sh); use sbatch for anything that launches the pipeline (smoke_run.sh, one_sample_run.sh, run.sh). That launch script is a lightweight Nextflow driver that then submits each pipeline step as its own SLURM job.

What is Apptainer?

A container runtime for HPCs (the successor to Singularity). Each pipeline step runs inside a pre-built container that holds the exact tool versions (STAR, Salmon, FastQC, …), so you install nothing and results are reproducible. On Phoenix you load it with module load Apptainer/<version> and select it with -profile apptainer. Images are cached (NXF_APPTAINER_CACHEDIR) and bind-mount /scratchdata1 so jobs can read your work directory.

Can I resume a failed run?

Yes — this is one of Nextflow's best features. Keep the work/ directory and re-submit; the run scripts already add -resume, so sbatch scripts/run.sh picks up where it stopped. Nextflow re-uses cached results for steps that already completed and only re-runs what failed or changed. Resume needs work/ intact, so don't delete it until the run is finished. To resume a specific earlier session, use -resume <run-id> (find IDs with nextflow log).

My job is queued (PD). Is something wrong?

Usually not. PD in squeue -u $USER means SLURM is waiting for resources — normal on a shared cluster, especially for large-memory steps on highmem/icelake. Read the reason in the last column: Priority or Resources means simply waiting your turn; QOSMax…/AssocMax… means you have hit an account or partition limit; ReqNodeNotAvail/PartitionTimeLimit means you asked for more time or resources than the partition allows. The Nextflow driver keeps running and submits steps as capacity frees up, so leave it alone. Only investigate if a job is pending far longer than usual or the reason points to a limit.

Why do I need a one-sample run?

It is a cheap dress rehearsal on your data, genome, and config. The tiny smoke test only proves Nextflow/Apptainer/SLURM work; a one-sample run catches the real-world problems — wrong strandedness, a mismatched GTF/GFF, an under-sized memory or partition request, a missing reference index, container/PATH issues — before you commit hours and a large allocation to every sample. If one real sample finishes cleanly, the full run almost always will too.

Where are my results?

Under the --outdir you set (the guide uses results/). Open multiqc/star_salmon/multiqc_report.html first for the overall QC summary. The count matrices you feed into DESeq2/edgeR are in star_salmon/ (e.g. salmon.merged.gene_counts.tsv); per-sample Salmon quant and genome-aligned BAMs are there too. Run provenance — software versions, params, and the timeline/report/trace — is in pipeline_info/.

Can I delete the work/ directory?

Yes, but only after the run has finished and you have confirmed the results in --outdir are complete. work/ holds Nextflow's intermediate files and is what makes -resume possible, so deleting it means any later resume starts from scratch. Clearing it (kept on /scratchdata1) is the main way to reclaim space. Caution: if the run used --publish_dir_mode symlink, files in results/ are links into work/ — deleting work/ breaks them, so copy the results out first or re-publish with copy.

Which aligner should I choose?

STAR + Salmon (star_salmon) is the recommended default: STAR aligns reads to the genome and Salmon quantifies, giving you both expression matrices and genome-aligned BAMs (useful for IGV, coverage tracks, or later SNP calling). Choose Salmon-only (--pseudo_aligner salmon --skip_alignment) when you just need fast transcript/gene quantification and do not need BAMs — it is lighter and avoids STAR's large memory footprint, which matters for big genomes.

What should I keep after the run?

Keep the count matrices (star_salmon/*gene_counts* and transcript counts), the MultiQC report, and pipeline_info/ (versions, params, trace) — these are what you need to analyse and to reproduce the run. Also keep your samplesheet.csv, configs/, nextflow.config, the scripts, and reference metadata. BAMs are large — keep them only if you will do alignment-based work (coverage, variant calling). Delete work/ once results are verified.

Phoenix specifics: modules, offline mode, and partitions

The run scripts load Phoenix modules (e.g. module load Nextflow/25.10.2 and module load Apptainer/1.2.5-GCCcore-12.3.0 — check module avail for current versions) and use -profile apptainer. Phoenix compute nodes have no internet, so the full and one-sample runs use -offline and NXF_OFFLINE=true; pre-pull the pipeline once on a login node with nextflow pull nf-core/rnaseq -r 3.26.0 before submitting. The smoke test downloads tiny test data and so needs internet. Partitions: jobs default to batch; memory-heavy steps for large genomes (STAR index, Salmon index/quant, STAR align) should go to highmem or icelake — see the commented large-genome block in nextflow.config. Containers run with --bind /scratchdata1 and a cache at NXF_APPTAINER_CACHEDIR; keep work/ on /scratchdata1 to avoid /hpcfs quota issues.

A step failed with “apptainer: No such file or directory” (exit 127)

SLURM child jobs sometimes start without the Apptainer module's bin on PATH. Set process.clusterOptions in nextflow.config to export the path (there is a commented example in the file); find the exact path with module show Apptainer/<version>.
