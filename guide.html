🧬 nf-core RNA-seq Runner — Product Guide v1

A template-based local-to-HPC runner for preparing inputs, validating infrastructure, launching nf-core/rnaseq, monitoring progress, and downloading results in 7 simple steps.

Product Goal

This is not just a GUI for launching nf-core. It is a standardized RNA-seq runner that guides users from local project preparation to HPC execution with validation, automatic recovery, monitoring, and result download.

Core value

  • Standardized project template
  • Automatic samplesheet generation
  • Input validation before upload
  • HPC environment and resource validation
  • Automatic safe fixes and resume
  • Simple execution workflow
  • Easy result retrieval

Workflow and Execution Context

  1. Initial Project — local
  2. Prepare & Validate Inputs — local
  3. Pipeline Configure — local
  4. Upload to HPC — local
  5. Preflight Validation — HPC
  6. Run & Monitor — HPC
  7. Download Results — local

Configuration Philosophy

Use one configuration file only: configs/configure.yaml.

Do not use params.yaml, nextflow_hpc.config, or resource_overrides.config in v1. Keep configuration simple.

Example fields

  • local_project_dir, hpc_user, hpc_host, hpc_project_dir
  • fastq_dir, samplesheet, reference, annotation
  • aligner, profile
  • memory, cpu, walltime
  • workdir, outdir, cache_dir

Step 1 — Initial Project

Purpose: Create a new project from a ready-to-use template.

User action: Download or clone template.

The template already contains scripts, configs, reports, folder structure, and README. Users should not manually create scripts.

Step 2 — Prepare & Validate Inputs

Runs locally.

Scripts

make_samplesheet.sh

input_validate.sh

Workflow

Add FASTQ files → Add reference genome → Add annotation → Run make_samplesheet.sh → Review samplesheet → Run input_validate.sh

Validation question

Are the project files correct?

Output

reports/input_validation_report.md

make_samplesheet.sh responsibilities

  • Scan input/fastq/
  • Detect paired-end FASTQ files
  • Match R1/R2 pairs
  • Infer sample names
  • Generate samplesheet.csv
  • Flag unmatched FASTQ files
  • Do not overwrite an existing samplesheet unless explicitly requested

input_validate.sh responsibilities

  • Check FASTQ files exist and paired-end files match
  • Check sample IDs are unique and samplesheet format is valid
  • Check reference, annotation, and required inputs exist
  • Estimate upload size and detect missing files

Step 3 — Pipeline Configure

Runs locally.

Script: configure.sh

Purpose: Choose how nf-core/rnaseq will run.

For v1, keep configuration very simple. The main required choice is the aligner.

Aligner options

  • STAR + Salmon — default
  • Salmon only

Advanced settings

Hidden by default. May include memory, CPU, walltime, workdir, outdir, scratch path, cache path, HPC project path, and profile.

Store all settings in configs/configure.yaml.

Step 4 — Upload to HPC

Runs locally.

Script: upload_to_hpc.sh

Purpose: Transfer the prepared project to HPC.

Responsibilities

  • Read HPC destination from configure.yaml
  • Upload project using rsync
  • Use resume-safe transfer
  • Create upload log
  • Verify transferred files when possible

Upload

scripts/, configs/, input/, reports/, and README.

Do not upload

Local cache, temporary files, old work directories, or unnecessary outputs.

Validation question

Has the project been successfully transferred to HPC?

Step 5 — Preflight Validation

Runs on HPC.

☑ Allow automatic fixes and resume during validation

Safe infrastructure and resource fixes may be applied automatically. Biological settings are never changed automatically.

Smoke Test

smoke_run.sh

Purpose: Validate the HPC environment using a tiny built-in test dataset.

Checks

  • Nextflow
  • Apptainer/Singularity
  • Container downloads
  • Cache configuration
  • Permissions, modules, work directory, scratch directory

Auto-fix examples

  • Container download failures
  • Missing cache directory or bad cache path
  • Permission issue
  • Module issue
  • Common Nextflow problems

Behaviour: Detect → Auto-fix → Resume

Output: reports/smoke_test_report.md

Validation question: Can this HPC + nf-core environment run?

One Sample Run

one_sample_run.sh

Purpose: Validate the current project configuration and estimate real resource requirements.

Workflow: Dry Run → Run one real sample.

Default sample: Largest sample, because resource validation should detect STAR memory issues, BEDTOOLS memory issues, walltime limits, scratch/work directory problems, disk quota issues, and bottlenecks.

Advanced options

  • Largest sample — default
  • Smallest sample — quick validation
  • User-selected sample

Auto-fix examples

  • STAR memory failure or CPU/thread mismatch
  • BEDTOOLS memory failure
  • Walltime issues
  • Quota or TMPDIR issues
  • Scratch recommendations
  • Resumable task failures

Output: reports/one_sample_validation_report.md

Validation question: Can this current project configuration survive a full run?

Never auto-change

Reference genome, annotation, strandedness, aligner choice, metadata, sample grouping, or experimental design.

Step 6 — Run & Monitor

Runs on HPC.

run.sh

run.sh

Launch the full nf-core/rnaseq workflow, read configure.yaml, and support Nextflow resume.

monitor_run.sh

monitor_run.sh

Show current stage, job status, failed tasks, disk usage, and recent logs. Make it clear where the pipeline is running or stuck.

Known issues should include suggested actions.

Step 7 — Download Results

Runs locally.

Script: download_results.sh

Purpose: Transfer final results from HPC back to local storage using rsync.

Features

  • Resume-safe transfer
  • Download log
  • Verification when possible

Download by default

results/, reports/, logs/, MultiQC reports, quantification outputs, and pipeline summaries.

Do not download by default

work/ or large BAM files. Large BAM files require explicit user selection.

Validation Hierarchy

  • Input Validation: Are the project files correct?
  • Smoke Test: Can this HPC + nf-core environment run?
  • One Sample Run: Can this current project configuration survive a full run?

Design Principles

  • One template
  • One configuration file
  • One main run script
  • Seven simple workflow steps
  • Local validation before upload
  • HPC validation before execution
  • Resource validation before full run
  • Automatic safe fixes and automatic resume
  • Minimal user decisions
  • Default aligner: STAR + Salmon
  • Default validation sample: Largest sample
  • Advanced settings hidden by default
  • Focus on preventing failed full-scale HPC runs
